# CLAUDE.md — Quantum Workbench

## Project Purpose

This is a tutorial-style course in quantum mechanics aimed at the author's own
learning and at advanced undergraduates / entry-level graduate students in
science and engineering. The course pairs conceptual understanding with
computational capability, building toward molecular quantum chemistry using
[PySCF](https://pyscf.org/).

The author (qqing) is the course designer and a co-learner. Claude's role is
**collaborator and co-author**: help design lessons, write exposition, implement
solvers, generate figures, and ensure physical and mathematical correctness. Do
not dumb things down — the audience is technically capable — but always make the
reasoning chain explicit.

---

## Pedagogical Principles

Every lesson follows the arc: **Concept -> Math -> Code -> Validation**.

### The Translation Layer
The bridge between mathematical formulation and numerical computation is treated
as a first-class skill. Units, discretization, dimensional analysis, and
sanity-checking are never afterthoughts. Every numerical result must be connected
back to a physical prediction or analytical limit.

### Core values
- **Build up gradually.** Start with analytically solvable systems, validate
  tools, then tackle problems that require numerical methods.
- **Good design over complexity.** Clean, readable code that maps directly to
  the physics. No black boxes, no premature abstraction.
- **Estimate before computing.** Dimensional analysis and order-of-magnitude
  estimates come before any code runs.
- **Validate ruthlessly.** Compare to analytical solutions, check
  orthonormality, count nodes, verify convergence rates, test limiting cases.
- **Physics constrains numerics.** Symmetry of the Hamiltonian dictates the
  eigensolver; unitarity constrains time-stepping methods. Always explain *why*
  a particular numerical choice is made.

## Lesson Structure

Each lesson lives in its own directory: `lesson-XX-topic-name/`

### Directory layout
```
lesson-XX-topic-name/
    lesson-XX.ipynb          # Main lesson notebook
    <module>.py              # Standalone reusable solver/utilities
    figures/                 # Saved plots (generated by notebook)
```

### Notebook sections (follow this template)
```
# Lesson N: Title
## From [starting concept] to [ending capability]

Goal / What you will learn / Prerequisites

---
## Part A: Conceptual Foundation
   Build physical intuition and mathematical framework.
   Introduce new physics concepts needed for the lesson.

## Part B: From Continuous Equation to Computable Numbers
   Dimensional analysis, choice of units, discretization strategy.
   The translation layer: map every math step to a code step.

## Part C: Building the Solver
   Implementation with a code-to-physics traceability table.
   Run the solver on the primary problem.

## Part D: Validation — Trust but Verify
   Compare to analytical results. Convergence studies.
   Sanity checks (orthonormality, node counts, symmetry, limiting cases).

## Part E: What If? — Beyond Analytical Solutions
   Apply the solver to problems without closed-form answers.
   Demonstrate the payoff of having a validated numerical tool.

## Summary
   What we built. Key takeaways (translation-layer lessons).
   What comes next (connect to future lessons).

## Exercises
   Graduated difficulty. Mix of: vary parameters, implement extensions,
   prove/derive something, connect to real physical systems.
```

Not every lesson must slavishly follow this template — adapt as the material
demands — but the arc (concept -> math -> code -> validate -> explore) should
always be present.

## Code Conventions

### Units
- **Atomic units throughout** (hbar = m_e = e = 1). Lengths in bohr, energies
  in hartree. This is what PySCF uses internally.
- Define conversion constants at the top of each module:
  ```python
  BOHR_PER_ANGSTROM = 1.8897259886
  BOHR_PER_NM = 18.8973
  HARTREE_TO_EV = 27.211386245988
  ```
- Always label axes and print statements with units.

### Style
- Python 3.8+ compatible.
- NumPy/SciPy/Matplotlib are the core stack. PySCF added when the course
  reaches molecular systems.
- Prefer explicit, readable code over clever one-liners. Every line should map
  to a physics or math concept from the exposition.
- Use classes to bundle system definition + grid + solutions (like
  `QuantumSystem1D`). Keep them lightweight — hold state, expose clean methods.
- Standalone `.py` modules should be importable and independently usable, with
  a module docstring showing a minimal usage example.
- Use `scipy.linalg.eigh` (not `eig`) for Hermitian problems. Always explain
  physics reasons for numerical choices.
- Normalize wavefunctions on the grid: `sum(|phi|^2) * dx = 1`.

### Plotting
- Save figures to `figures/` within the lesson directory.
- Use `dpi=150, bbox_inches='tight'` for saved figures.
- Label axes with units. Include legends with state indices and energies.
- For eigenstate plots: option to offset wavefunctions by their energy.

### Notebooks
- Markdown cells carry the full conceptual and mathematical exposition.
  They are not afterthought captions — they are the lesson text.
- Code cells should be short and focused. Each cell does one thing.
- Print numerical results in aligned, labeled tables.
- Use LaTeX math extensively in markdown cells.

## Mathematical and Physics Notation

- Operators wear hats: $\hat{H}$, $\hat{p}$
- Wavefunctions: $\psi(x,t)$ for time-dependent, $\phi(x)$ for
  time-independent eigenstates
- Bra-ket when appropriate: $\langle \phi_m | \phi_n \rangle$
- Eigenvalue equations: $\hat{H}\phi_n = E_n \phi_n$
- Always state what each symbol means in a table or inline when first introduced
- Use boxed equations for the key results: `$$\boxed{...}$$`

---

## Writing Style for Exposition

- **Direct and precise.** No filler. Every sentence should teach something or
  set up the next idea.
- **Connect abstractions to concrete examples.** When introducing a general
  principle (e.g., exponentials from infinitesimal steps), ground it in
  multiple familiar contexts before applying it to quantum mechanics.
- **Flag what can be skipped.** Use notes like *(This section can be skipped on
  first reading.)* for deeper/advanced material so the reader can control depth.
- **Anticipate confusion.** Where the math notation or physics convention is
  likely to trip someone up, pause and clarify explicitly.
- **Cross-reference between lessons.** "We saw in Lesson N that..." and "We will
  revisit this in Lesson N when..." to build a connected narrative.
- **Use comparison tables** to contrast related concepts (e.g., diffusion vs.
  Schrodinger, classical vs. quantum).
- **Boxed key results.** The most important equations get `\boxed{}`.
- **Habits and design principles in blockquotes.** Use `>` blockquotes to
  highlight recurring methodological lessons (e.g., "Always estimate before
  computing").

## PySCF Integration Roadmap

The course builds from first-principles solvers toward PySCF. The progression:

1. **Foundations (Lessons 1-3ish):** Custom solvers for 1D/2D/3D toy systems.
   Build intuition for eigenvalue problems, basis representations,
   convergence, and the variational principle. No PySCF yet.

2. **Bridge lessons:** Introduce basis sets (from grid -> Gaussian basis
   functions), the self-consistent field concept, and the Hartree-Fock
   approximation — first implemented from scratch on toy systems, then
   connected to what PySCF does under the hood.

3. **PySCF lessons:** Use PySCF for real molecular calculations (HF, DFT, MP2,
   CCSD, CASSCF, etc.). The student already understands what the code is
   doing because they built simpler versions.

### PySCF specifics to keep in mind
- PySCF uses atomic units internally — consistent with our convention.
- PySCF is modular (SCF -> post-HF methods layer naturally). The course
  structure should mirror this: build the SCF concept first, then layer
  correlation methods on top.
- Key PySCF modules relevant to the course roadmap:
  - `pyscf.gto` — Gaussian basis sets and molecular geometry
  - `pyscf.scf` — Hartree-Fock (RHF, UHF, ROHF) and DFT (RKS, UKS)
  - `pyscf.mp` — Moller-Plesset perturbation theory (MP2)
  - `pyscf.cc` — Coupled cluster (CCSD, CCSD(T), EOM-CCSD)
  - `pyscf.mcscf` — Multi-configurational SCF (CASSCF, CASCI)
  - `pyscf.tddft` — Time-dependent DFT for excited states
  - `pyscf.fci` — Full configuration interaction
- When introducing PySCF, always show the connection to the concepts the
  student has already implemented: "Remember our Hamiltonian matrix from
  Lesson 1? PySCF builds the same object, but in a Gaussian basis instead of
  on a grid."

## Things to Avoid

- **Black-box calls.** Never use a library function without explaining what it
  does and why it is the right choice.
- **Skipping validation.** Even for "obvious" results, run the checks.
- **SI unit arithmetic.** Avoid juggling factors of 10^-34 in code — use atomic
  units and convert only for final display.
- **Over-engineering.** A 60-line solver class is fine. Don't build a framework
  when a script will do.
- **Disconnected code.** Every code cell should be preceded by exposition that
  explains what it does and why. Never drop code without context.
- **Assuming knowledge.** When a concept from linear algebra, calculus, or
  classical mechanics is used, briefly state what it is, even if the audience
  "should" know. A one-sentence reminder costs nothing.
